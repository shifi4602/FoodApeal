## 🎯 Project Overview

**FoodAppeal** is an online food shop platform. Customers can:

- 🛍️ Browse and filter products by category, price range, and description
- 🛒 Add items to a shopping basket and place orders
- 👤 Register / log in and view their order history
- 🗺️ Find restaurant branches on an interactive map
- 🍳 Explore cooking and hosting services
- 🎁 Browse gift options

**Admins** can:

- ➕ Add new products with image upload
- ✏️ Edit existing product details
- 📋 Manage orders and view all users

## ⚙️ Backend

The backend is a clean, layered **ASP.NET Core Web API** following the **separation of concerns** principle. Each layer has a single responsibility and communicates only with the layer directly below it.

```
Controller → Service → Repository → EF Core → SQL Server
```

---

### Architecture & Layers

```
backend/
├── Entities/         🗄️  Domain models (EF Core entities)
├── DTOs/             📦  Data Transfer Objects
├── Repositories/     🔌  Data access (EF Core + LINQ)
├── Services/         🧠  Business logic
└── WebApiShop/       🌐  ASP.NET Core app (Controllers, Middleware, Program.cs)
```

Each project is a separate class library with clearly defined interfaces — making layers swappable and testable in isolation.

---

### Entities (Database Models)

Entities represent the actual database tables, auto-generated by **EF Core Power Tools** and stored in the `Entities` project.

| Entity       | Key Fields                                                       | Relationships                              |
|--------------|------------------------------------------------------------------|--------------------------------------------|
| 🛒 `Product`  | `Id`, `Name`, `Description`, `Price`, `ImageUrl`, `IsAvailable` | Belongs to `Category`, has many `OrderItems` |
| 👤 `User`     | `Id`, `Email`, `FirstName`, `LastName`, `Password`, `IsAdmin`   | Has many `Orders`                          |
| 📦 `Order`    | `Id`, `UserId`, `OrderDate`, `OrderSum`, `Status`               | Belongs to `User`, has many `OrderItems`   |
| 🧾 `OrderItem`| `Id`, `OrderId`, `ProductId`, `Quantity`                        | Belongs to `Order` and `Product`           |
| 🗂️ `Category` | `Id`, `Name`                                                    | Has many `Products`                        |
| 📊 `Rating`   | `Host`, `Method`, `Path`, `UserAgent`, `RecordDate`             | Standalone — one row per HTTP request      |

> The `Rating` entity records every HTTP request that passes through the app (host, method, path, timestamp). This acts as an **audit log / traffic tracker**.

---

### DTOs (Data Transfer Objects)

DTOs live in the `DTOs` project and are used to decouple the API contract from internal database entities. They prevent over-posting, hide sensitive fields, and shape the data for the client.

All DTOs are defined as C# **records** (immutable by default):

| DTO                  | Direction         | Key Fields                                                    |
|----------------------|-------------------|---------------------------------------------------------------|
| `ProductDTO`         | Server → Client   | All product fields + `CategoryName` (flattened)               |
| `PostProductDTO`     | Client → Server   | Product fields (no `CategoryName`, `IsAvailable` defaults to `true`) |
| `OrderDTO`           | Both              | Order fields + nested `ICollection<OrderItemDTO>`             |
| `UserDTO`            | Server → Client   | User fields + nested `ICollection<OrderDTO>`                  |
| `PostUserDTO`        | Client → Server   | Registration fields                                           |
| `LoginUserDTO`       | Client → Server   | `Email`, `Password`                                           |
| `UpdateUserDTO`      | Client → Server   | Editable user fields                                          |
| `CategoryDTO`        | Server → Client   | `Id`, `Name`                                                  |
| `PageResponseDTO<T>` | Server → Client   | `Data`, `TotalItems`, `CurrentPage`, `PageSize`, `HasPreviousPage`, `HasNextPage` |

> 📌 **Why two Product DTOs?** `ProductDTO` includes `CategoryName` (resolved server-side), while `PostProductDTO` only needs the `CategoryId`. This avoids sending unnecessary data in requests and keeps responses rich.

---

### Entity Framework Core

**EF Core** is used as the ORM. The `ApiDBContext` (in `Repositories`) exposes a `DbSet<T>` for every table: `Categories`, `Orders`, `OrderItems`, `Products`, `Ratings`, and `Users`.

Relationships are configured with the **Fluent API** in `OnModelCreating` — for example, `Order → User` (many-to-one) and `OrderItem → Order / Product` (many-to-one each).

The connection string is stored in `appsettings.json` and injected at startup:

```csharp
builder.Services.AddDbContext<ApiDBContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("School")));
```

---

### Repositories Layer

Each entity has a dedicated repository that implements an interface. This pattern makes the data access logic mockable in tests.

Every repository follows this interface shape (example: `IProductRepository`):

```csharp
Task<List<Product>> GetProducts(int position, int skip, int?[] categoryIds,
    string? description, int? maxPrice, int? minPrice);
Task<Product> GetProductById(int id);
Task<Product> AddProduct(Product product);
Task UpdateProduct(int id, Product product);
```

The `GetProducts` implementation builds a single LINQ query that handles **filtering by category, description, price range** AND **pagination** — all translated by EF Core into one optimized SQL query. Results are eager-loaded with `.Include(product => product.Category)`.

---

### Services Layer

Services contain the **business logic**. They receive DTOs from controllers, interact with repositories, and use AutoMapper to convert between entity and DTO types.

Each service method follows the same pattern: receive a DTO → map to entity → call repository → map result back to DTO:

```csharp
public async Task<UserDTO> AddUser(PostUserDTO user)
{
    return _mapper.Map<User, UserDTO>(
        await _userRepository.AddUser(_mapper.Map<PostUserDTO, User>(user))
    );
}
```

Additional business rules enforced in the service layer:
- 🔒 **Password strength** — minimum Zxcvbn score of 2 required to register
- 📧 **Unique email** — duplicate emails are rejected before saving

All services are registered as **scoped** dependencies in `Program.cs`.

---

### Controllers (API Layer)

Controllers are thin — they delegate all logic to services and only handle HTTP concerns (routing, status codes, request/response shaping).

Example request: `GET /api/products?categoryIds=1&description=pizza&minPrice=5&maxPrice=50&skip=8&position=1`

| Controller           | Endpoints                                      |
|----------------------|------------------------------------------------|
| 🛒 `ProductsController`  | GET (all/filter), GET (by id), POST, PUT   |
| 🗂️ `CategoriesController`| GET all                                    |
| 📦 `OrdersController`    | GET (all), GET (by id), POST, PUT          |
| 👤 `UsersController`     | GET (all), GET (by id), POST, POST /login  |
| 🔑 `PasswordsController` | POST (validate strength)                   |
| 🖼️ `UploadController`    | POST /upload (multipart file)              |

---

### AutoMapper

AutoMapper is configured in `WebApiShop/AutoMapper.cs` as a `Profile`. It defines bidirectional mappings between all entities and DTOs using `.ReverseMap()`, which allows a single declaration to map in both directions (entity → DTO and DTO → entity).

Mapped pairs: `Product ↔ ProductDTO`, `Product ↔ PostProductDTO`, `Order ↔ OrderDTO`, `OrderItem ↔ OrderItemDTO`, `User ↔ UserDTO / PostUserDTO / UpdateUserDTO / LoginUserDTO`, `Category ↔ CategoryDTO`.

---

### Middleware

Two custom middleware components are registered in the pipeline:

#### 🛡️ ErrorHandlingMiddleware

Wraps every request in a `try/catch`. If an unhandled exception occurs, it returns `HTTP 500` and logs the full stack trace via NLog.

#### 📈 RatingMiddleware

Before passing the request to the next handler, it records the host, method, path, referer, user-agent, and timestamp to the `Ratings` table — providing a complete request audit log.

**Middleware pipeline order:**

```csharp
app.UseCors("AllowAngular");
app.UseHttpsRedirection();
app.UseErrorHandling();   // 🛡️ global error handler
app.UseRating();          // 📈 request logger
app.UseStaticFiles();
app.UseAuthorization();
app.MapControllers();
```

---

### Logging with NLog

NLog is wired up via `builder.Host.UseNLog()` and configured in `nlog.config`. All unhandled exceptions are captured by `ErrorHandlingMiddleware` and written to the configured NLog targets (file, console, etc.). Controllers that need logging receive an `ILogger<T>` via constructor injection.

---

### Password Validation

Password strength is validated using [**Zxcvbn**](https://github.com/zxcvbn-ts/zxcvbn) — the same algorithm used by Dropbox. It returns a score from `0` (very weak) to `4` (very strong). A minimum score of **2** is required to register. If the password is too weak, the API returns `400 Bad Request`.

---

### File Upload

`UploadController` accepts a multipart file, saves it to `wwwroot/images/` with a GUID-based filename (to avoid collisions), and returns the public URL. That URL is then stored in the product's `ImageUrl` field and served as a static file.

---

### CORS Configuration

CORS is configured to allow requests only from `http://localhost:4200` (the Angular dev server), accepting any header and method.

### State Management with Signals

The app uses Angular's built-in **Signals** for reactive state. Each service holds a private `signal<T>` and exposes it as read-only. Components read signals directly in templates — Angular re-renders only when the value changes.

```typescript
private productsSignal = signal<Product[]>([]);
getProducts() { return this.productsSignal.asReadonly(); }
```

| Service               | Responsibility                                          |
|-----------------------|---------------------------------------------------------|
| `UserService`         | Login, register, current user state, order history      |

All services call the backend via `HttpClient` using the base URL from `environment.ts`. The `apiUrl` is the only thing that changes between environments.

---

### Route Guards

The `adminGuard` protects admin-only routes. It reads the current user from `UserService` and redirects to `/login` if `user.isAdmin` is false. Applied to `/admin` and `/product-edit/:id`.
